## Summary

Phase 10 adds passive monitoring of Claude Code tool invocations and session lifecycle events. The core mechanism is Claude Code's built-in hook system: when a hook fires, Claude Code sends a JSON payload to a configured command or HTTP endpoint. The harness runs a FastAPI server on localhost:8099 as an asyncio task alongside CLIRuntime.run(). Claude Code POSTs hook events directly to that server via the native `type: "http"` hook configuration — eliminating the curl relay script originally planned.

The most important discovery is that **Claude Code has a native `type: "http"` hook** (added to the documented API). This is preferable to the curl relay approach because it is pure configuration (zero shell scripts), which is a strictly stronger security posture than even a read-only relay script. The `~/.claude/settings.json` file (global user settings) is generated by the Manager at spawn time and mounted read-only into the container. Claude Code loads this file in all modes including `-p` headless mode (it only skips it with `--bare`, which the harness does not use).

The audit trail uses Redis sorted sets (`audit:{task_id}`) with UNIX timestamp scores, a 24-hour EXPIRE on the key, and structured JSON stdout mirroring. The Gateway adds a `GET /tasks/{task_id}/audit` route following the established pattern. CLIRuntime already has `_post_progress()` and `_publish_state()` — Stop hook events use these directly to emit task_progress lifecycle events.

**Primary recommendation:** Use `type: "http"` in the settings.json hook config pointing at `http://127.0.0.1:8099/hooks`. Skip the curl relay script. Run the FastAPI hook server as `asyncio.create_task(uvicorn.Server(config).serve())` inside `CLIRuntime.run()`.

### Pattern 1: FastAPI Embedded in asyncio via uvicorn.Server.serve()

**What:** Run a second FastAPI HTTP server as an asyncio task inside the existing CLIRuntime event loop. Uses `uvicorn.Config` + `uvicorn.Server` directly — NOT `uvicorn.run()` which internally calls `asyncio.run()` and cannot be used inside an already-running loop.

**When to use:** Any time a secondary HTTP server must share the same event loop as other async code without blocking the event loop or requiring a separate process.

**Example:**

```python
# Source: https://www.uvicorn.org/ — Config and Server Instances section
import uvicorn
from fastapi import FastAPI

app = FastAPI()

async def start_hook_server() -> uvicorn.Server:
    config = uvicorn.Config(
        app,
        host="127.0.0.1",
        port=8099,
        log_level="warning",
        access_log=False,
    )
    server = uvicorn.Server(config)
    asyncio.create_task(server.serve())
    return server
```

**Critical:** Use `uvicorn.Server(config).serve()` — NOT `uvicorn.run()`. The `run()` method calls `asyncio.run()` internally and raises `RuntimeError: This event loop is already running` when called from inside an async context.

**Shutdown:** `server.should_exit = True` stops the server gracefully. Call this in CLIRuntime.stop() alongside the PTY child shutdown.

### Pattern 2: Claude Code Native HTTP Hook Configuration

**What:** Configure Claude Code to POST hook event JSON directly to the harness HTTP endpoint using the native `type: "http"` hook. No shell scripts required.

**When to use:** Any Claude Code hook monitoring scenario. The HTTP type is available for all 21+ hook events including PostToolUse, Stop, SessionEnd, SubagentStop.

**The settings.json hook config (generated by Manager, mounted read-only at `~/.claude/settings.json`):**

```json
      {
        "hooks": [
          {
            "type": "http",
            "url": "http://127.0.0.1:8099/hooks",
            "timeout": 10
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "http",
            "url": "http://127.0.0.1:8099/hooks",
            "timeout": 10
          }
        ]
      }
    ],
    "SessionEnd": [
      {
        "hooks": [
          {
            "type": "http",
            "url": "http://127.0.0.1:8099/hooks",
            "timeout": 10
          }
        ]
      }
    ],
    "SubagentStop": [
      {
        "hooks": [
          {
            "type": "http",
```

**No matcher needed** for PostToolUse, Stop, SessionEnd, SubagentStop when you want all tools/events. Omitting `matcher` matches everything.

**Key insight:** Claude Code loads `~/.claude/settings.json` (global user settings) in all non-bare modes. The harness uses `claude -p ... --dangerously-skip-permissions` which is headless but not bare. Global user settings apply. The Manager writes to `~/.claude/settings.json` inside the container at spawn time before mounting it read-only.

**Wait** — there is a subtlety here: the Manager mounts the file at spawn time before the container starts. Inside the container, the path is `/root/.claude/settings.json` (container root user). The Manager generates the file on the host and mounts it read-only into the container.

### Pattern 4: Pydantic Discriminated Union for Hook Events

**What:** A single `POST /hooks` endpoint accepts all four event types. Use Pydantic's discriminated union (via `hook_event_name` literal field) to validate and route.

```python
# Source: Pydantic v2 discriminated unions + Claude Code hook schema (verified)
from typing import Literal, Union, Annotated
from pydantic import BaseModel, Field

class PostToolUseEvent(BaseModel):
    hook_event_name: Literal["PostToolUse"]
    session_id: str
    tool_name: str
    tool_use_id: str
    tool_input: dict
    tool_response: dict
    cwd: str = ""
    permission_mode: str = ""
    transcript_path: str = ""

class StopEvent(BaseModel):
    hook_event_name: Literal["Stop"]
    session_id: str
    stop_hook_active: bool = False
    last_assistant_message: str = ""
    cwd: str = ""
    permission_mode: str = ""
    transcript_path: str = ""

class SubagentStopEvent(BaseModel):
    hook_event_name: Literal["SubagentStop"]
    session_id: str
    stop_hook_active: bool = False
    agent_id: str = ""
    agent_type: str = ""
    agent_transcript_path: str = ""
    last_assistant_message: str = ""
    cwd: str = ""
    permission_mode: str = ""
    transcript_path: str = ""

class SessionEndEvent(BaseModel):
    hook_event_name: Literal["SessionEnd"]
    session_id: str
    reason: str = ""
    cwd: str = ""
    transcript_path: str = ""

class UnknownEvent(BaseModel):
    hook_event_name: str  # fallback for any future event types

HookEvent = Annotated[
    Union[PostToolUseEvent, StopEvent, SubagentStopEvent, SessionEndEvent, UnknownEvent],
    Field(discriminator="hook_event_name")
]
```

**Important:** Wrap in try/except at the route level. If Pydantic validation raises on any event, catch it, log WARNING, return 200. Never return 4xx that could cause Claude Code to treat the hook as an error. D-04 requires unknown types to return 200 and be discarded.

### Pattern 5: Redis Sorted Set for Audit Trail

**What:** ZADD with UNIX timestamp as score, JSON blob as member. EXPIRE the key after 24h.

```python
# Source: https://redis.readthedocs.io/en/stable/examples/asyncio_examples.html
import time
import json

async def _write_audit_entry(
    redis: aioredis.Redis,
    task_id: str,
    tool_name: str,
    success: bool,
) -> None:
    """Write one audit entry to Redis sorted set audit:{task_id}."""
    ts = time.time()
    entry = json.dumps({
        "tool_name": tool_name,
        "timestamp": ts,
        "success": success,
    })
    key = f"audit:{task_id}"
    # ZADD score=timestamp, member=json blob
    await redis.zadd(key, {entry: ts})
    # Reset 24h TTL on every write (key-level EXPIRE is the only option)
    await redis.expire(key, 86400)
```

**Query pattern for Gateway GET /tasks/{task_id}/audit:**
```python
entries_raw = await redis.zrange(f"audit:{task_id}", 0, -1)
# Returns list of JSON strings sorted by timestamp score
entries = [json.loads(e) for e in entries_raw]
```

**No pagination needed** for 24h-TTL data at KubexClaw's current scale. ZRANGE 0 -1 returns all entries sorted ascending by timestamp.

### Pattern 7: Hook Server Shutdown Integration

**What:** When CLIRuntime.stop() is called, also stop the uvicorn hook server.

```python
class CLIRuntime:
    def __init__(self, config):
        ...
        self._hook_server: uvicorn.Server | None = None

    async def run(self) -> None:
        ...
        if config.runtime != "openai-api":
            self._hook_server = await start_hook_server()
        await self._task_loop()

    def stop(self) -> None:
        self._running = False
        self._stop_event.set()
        if self._hook_server is not None:
            self._hook_server.should_exit = True  # graceful uvicorn shutdown
        if self._child is not None:
            asyncio.ensure_future(self._graceful_shutdown())
```

### Anti-Patterns to Avoid

- **`uvicorn.run()` inside async context**: Internally calls `asyncio.run()`, raises RuntimeError. Always use `uvicorn.Server(config).serve()` directly.
- **curl relay script when HTTP hook type exists**: Shell scripts have a surface area that read-only mounts do not fully eliminate (the script itself must be mounted). `type: "http"` is pure configuration — zero shell execution.
- **Writing settings.json to `.claude/settings.json` (project dir) instead of `~/.claude/settings.json` (global user dir)**: Claude Code's project directory `.claude/settings.json` can potentially be influenced by task content if the task operates in `/app`. Writing to the global `~/.claude/settings.json` and mounting it read-only is the correct security boundary.
- **Returning 4xx from the hook endpoint on unknown event types**: Claude Code treats non-2xx hook responses as errors and may log warnings or retry. Always return 200 for unknown event types (D-04).
- **Storing tool_input or tool_response content in audit entries**: Tool input/output can be megabytes (file content). D-14 mandates minimal audit footprint: tool_name, timestamp, success boolean only.
- **Calling `asyncio.gather()` on the hook server and task_loop directly**: The hook server runs as a fire-and-forget `asyncio.create_task()`, not awaited alongside the task loop. The task loop is what `run()` awaits. This avoids a race where both tasks complete simultaneously and the harness exits prematurely.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Hook event delivery to harness | curl relay shell script | `type: "http"` in settings.json | Native Claude Code feature; zero shell surface area; simpler and more secure |
| Async HTTP server embedded in asyncio | Custom asyncio socket server | `uvicorn.Server(config).serve()` as asyncio task | Uvicorn handles HTTP/1.1, ASGI, connection lifecycle correctly |
| JSON validation at hook endpoint | Hand-rolled type checking | Pydantic discriminated union on `hook_event_name` | Pydantic v2 handles discriminated unions natively; already in the project stack |
| Time-series audit storage | Redis list or custom DB | Redis sorted set with UNIX timestamp score | ZADD/ZRANGE/ZRANGEBYSCORE is built for this; O(log N) insert; range queries by time |
| Read-only file injection into container | Container image rebuild | Docker SDK `mode: ro` bind mount | Same pattern already used for config.yaml and skill files in lifecycle.py |

**Key insight:** The entire hook delivery mechanism is a settings.json configuration file — no scripts, no sockets, no custom relay. Claude Code does all the heavy lifting via its built-in HTTP hook type. The harness just needs to run a FastAPI server and handle the POST.

## Common Pitfalls

### Pitfall 1: settings.json Target Path — Project vs Global

**What goes wrong:** Writing the hook config to `/app/.claude/settings.json` (the project directory inside the container) instead of `/root/.claude/settings.json` (the global user settings). Claude Code loads settings from the project directory `.claude/settings.json` first — but this path can be influenced by task content if the task operates in `/app`. An injected task could create a competing `.claude/settings.json` in the working directory that overrides the mounted global config.

**Why it happens:** The project-level `.claude/settings.json` is co-located with the agent's workspace, which feels natural. But it is in a writable directory.

**How to avoid:** Write the hook config to `/root/.claude/settings.json` (global user level) and mount it read-only. This path cannot be shadowed by project-level settings because settings merge precedence is: local project > shared project > global — but the global path is what the Manager controls. More importantly, the global path is outside the `/app` working directory where task content could create files.

Wait — re-reading the settings precedence: `.claude/settings.local.json` > `.claude/settings.json` > `~/.claude/settings.json`. A malicious task could create `.claude/settings.local.json` in `/app/.claude/settings.local.json` and override the global config. The real defense is therefore D-02 (read-only mount) applied at the global level AND the container not having write access to `.claude/` directories at all.

**The correct defense (D-08):** Mount `/root/.claude/` as a whole read-only if possible, or at minimum mount `/root/.claude/settings.json:ro`. Since the credential file also lives in `/root/.claude/`, use a tmpfs or separate mount for credentials.

**Warning signs:** `docker exec <container> ls -la /root/.claude/settings.json` shows `rw` permissions.

### Pitfall 2: uvicorn.run() vs uvicorn.Server.serve()

**What goes wrong:** Calling `uvicorn.run(app, host="127.0.0.1", port=8099)` from inside `CLIRuntime.run()` which is already running inside `asyncio.run()`. This raises `RuntimeError: This event loop is already running` because `uvicorn.run()` internally calls `asyncio.run()`, which cannot be nested.

**Why it happens:** `uvicorn.run()` is the entry point documented in most examples. It is designed for top-level invocation, not embedding.

**How to avoid:** Always use `uvicorn.Config` + `uvicorn.Server` + `asyncio.create_task(server.serve())`. The `server.serve()` coroutine runs in the existing event loop without creating a new one.

**Warning signs:** `RuntimeError: This event loop is already running` in harness logs immediately after hook server startup.

### Pitfall 3: HTTP Hook Timeout Too Short Causes Claude Code Error Logs

**What goes wrong:** The hook handler is slow (Redis write + pub/sub publish together) and exceeds the `timeout` configured in settings.json. Claude Code logs a non-blocking error for HTTP hooks that time out and continues execution. However if the timeout is extremely short (< 1s) and Redis is cold-starting, every hook event generates an error log.

**Why it happens:** Default HTTP hook timeout is 30s, but D-07 suggests 10s. For an in-container loopback call to a FastAPI handler that does two Redis operations, 10s is more than sufficient under normal conditions. The risk is Redis unavailability at container start.

**How to avoid:** Keep `timeout: 10` (D-07 intent). Make the hook handler non-blocking: wrap all Redis operations in try/except and return 200 immediately even if Redis fails. The audit trail is best-effort; never let a Redis failure cause Claude Code to error. Return 200 within < 100ms in the happy path.

**Warning signs:** Claude Code logs `"hook_error"` events for every tool invocation in harness stdout.

### Pitfall 4: Audit Key Collision Between Multiple Concurrent Tasks

**What goes wrong:** CLIRuntime is READY/BUSY with one task at a time (no concurrent task execution). But if the task_id is `None` or `"unknown"` when a hook arrives (e.g., the hook fires before `_execute_task` sets the current task_id), all hook events write to `audit:unknown` — a shared collision key.

**Why it happens:** The hook server is a separate asyncio task that does not directly share state with `_task_loop`. The `task_id` of the currently-executing task must be tracked in a shared attribute on CLIRuntime (or a dedicated asyncio-safe shared state).

**How to avoid:** Add `self._current_task_id: str | None = None` to CLIRuntime. Set it at the start of `_execute_task(task)` (after the task is dequeued), clear it after task completion. The hook server (via the CLIRuntime reference passed to it) reads `_current_task_id`. If `None`, discard the event with a WARNING log rather than writing to `audit:None`.

**Warning signs:** Redis contains a key `audit:None` or `audit:unknown` accumulating all hook events.

### Pitfall 5: Hook Config File Writable at Runtime (CVE-2025-59536)

**What goes wrong:** `~/.claude/settings.json` is mounted read-write or is inside a named volume. A prompt injection causes Claude Code to attempt to write a modified settings.json with a malicious hook command, replacing the audit relay with an exfiltration script.

**Why it happens:** Named volumes are read-write by default. If the credential volume (`kubex-creds-{agent_id}`) covers `/root/.claude/` (which it does in the current lifecycle.py for claude-code), the settings.json would be inside that writable named volume.

**The conflict:** `CLI_CREDENTIAL_MOUNTS["claude-code"] = "/root/.claude"` (lifecycle.py line 49) mounts the entire `/root/.claude/` directory as a named volume. If settings.json is bind-mounted at `/root/.claude/settings.json:ro` AFTER the volume is mounted, the bind mount takes precedence over the volume for that specific file — this is how Docker mount overlaying works.

**How to avoid:** Bind-mount `/root/.claude/settings.json:ro` as a SEPARATE mount entry after the named volume mount. Docker processes mounts in order and the bind mount shadows the named volume for that specific file path. Verify with `docker exec <container> stat /root/.claude/settings.json` showing read-only permissions despite the parent directory being a writable volume.

**Warning signs:** `docker exec <container> touch /root/.claude/settings.json` succeeds.

### Pitfall 6: Hook Events Arriving Before task_id is Set

**What goes wrong:** Claude Code may fire a hook event within milliseconds of the `-p` process starting (before `_execute_task` has set `_current_task_id`). If `PostToolUse` fires before the harness associates the hook event with a task_id, the audit entry is lost.

**Why it happens:** The PTY child is spawned, Claude Code initializes, and may immediately call a tool (e.g., a filesystem scan) before Python's asyncio event loop has processed the `_current_task_id = task_id` line in `_execute_task`.

**How to avoid:** Set `self._current_task_id` immediately when the task is dequeued (before `_run_cli_process` is called). This is synchronous Python assignment before any await — it happens atomically in the event loop before the hook server's next iteration can read it.

```python
    tool_response: dict = Field(default_factory=dict)

    permission_mode: str = ""

    hook_event_name: Literal["SubagentStop"]
    session_id: str
    stop_hook_active: bool = False
    agent_id: str = ""
    agent_type: str = ""
    agent_transcript_path: str = ""
    last_assistant_message: str = ""
    cwd: str = ""
    permission_mode: str = ""

    hook_event_name: Literal["SessionEnd"]
    session_id: str
    reason: str = ""
    cwd: str = ""
    transcript_path: str = ""

# Discriminated union — Pydantic v2 syntax
HookEvent = Annotated[
    Union[PostToolUseEvent, StopEvent, SubagentStopEvent, SessionEndEvent],
    Field(discriminator="hook_event_name")
]


def create_hook_app(cli_runtime: Any) -> FastAPI:
    """Create the FastAPI hook app wired to a CLIRuntime instance."""
    app = FastAPI(title="kubex-hook-server", docs_url=None, redoc_url=None)

    @app.post("/hooks")
    async def receive_hook(request: Request) -> JSONResponse:
        raw = await request.json()
        event_name = raw.get("hook_event_name", "unknown")

        try:
            event = HookEvent.model_validate(raw)  # type: ignore[arg-type]
        except (ValidationError, Exception):
            logger.warning("hook_validation_failed event_type=%s", event_name)
            return JSONResponse(status_code=200, content={"ok": True})

        # Route by type — discriminated union already parsed
        if isinstance(event, PostToolUseEvent):
            await cli_runtime._on_post_tool_use(event)
        elif isinstance(event, StopEvent):
            await cli_runtime._on_stop(event)
        elif isinstance(event, SubagentStopEvent):
            await cli_runtime._on_subagent_stop(event)
        elif isinstance(event, SessionEndEvent):
            await cli_runtime._on_session_end(event)

        return JSONResponse(status_code=200, content={"ok": True})

    return app


async def start_hook_server(cli_runtime: Any) -> uvicorn.Server:
    """Start the hook server as an asyncio task. Returns the server instance."""
    app = create_hook_app(cli_runtime)
    config = uvicorn.Config(
        app,
        host="127.0.0.1",
        port=8099,
        log_level="warning",
        access_log=False,
    )
    server = uvicorn.Server(config)
    asyncio.create_task(server.serve())
    return server
```

```python
    logger.info(

        task_id, event.tool_name, success
    )

async def _on_stop(self, event: StopEvent) -> None:
    task_id = self._current_task_id
    if task_id is None:
        return
    # D-15: emit task_progress on pub/sub
    await self._post_progress(
        task_id,
        content=f"stop: {event.last_assistant_message[:200]}",
        final=False,
    )

async def _on_subagent_stop(self, event: SubagentStopEvent) -> None:
    # D-17: audit log only, no pub/sub propagation
    logger.info(
        "subagent_stop session=%s agent_type=%s",
        event.session_id, event.agent_type
    )

async def _on_session_end(self, event: SessionEndEvent) -> None:
    # D-16: metadata enrichment only
    logger.info(
        "session_end session=%s reason=%s",
        event.session_id, event.reason
    )

async def _write_audit_entry(self, task_id: str, tool_name: str, success: bool) -> None:
    if self._redis is None:
        return
    ts = time.time()
    entry = json.dumps({"tool_name": tool_name, "timestamp": ts, "success": success})
    key = f"audit:{task_id}"
    try:
        await self._redis.zadd(key, {entry: ts})
        await self._redis.expire(key, 86400)  # 24h TTL
    except Exception:
        pass  # audit trail must never block task processing
```

```python
from pydantic import BaseModel

class AuditEntry(BaseModel):
    tool_name: str
    timestamp: float
    success: bool

@router.get("/tasks/{task_id}/audit")
async def get_task_audit(task_id: str, request: Request) -> JSONResponse:
    """Return the audit trail for a task from Redis sorted set."""
    gateway: GatewayService = request.app.state.gateway_service

    if gateway.redis_db0 is None:
        return JSONResponse(status_code=503, content={"error": "Redis unavailable"})

    key = f"audit:{task_id}"
    try:
        raw_entries = await gateway.redis_db0.zrange(key, 0, -1)
    except Exception:
        return JSONResponse(status_code=500, content={"error": "Redis error"})

    entries = []
    for raw in raw_entries:
        try:
            entries.append(json.loads(raw))
        except Exception:
            pass

    return JSONResponse(content={"task_id": task_id, "entries": entries})
```

**Deprecated/outdated:**

However since D-09 is a locked decision about "static curl pipe script", the planner should note this as an upgrade opportunity (the D-09 curl approach is valid but suboptimal given what's now available). ## Open Questions

1. **D-09 vs type: "http" conflict**
   - What we know: D-09 locked "curl pipe script, read-only mount" as the relay mechanism. The `type: "http"` hook now achieves exactly the same result without any shell script.
   - What's unclear: Whether to use D-09 (curl script) as designed, or upgrade to `type: "http"` (no script needed). The security posture of `type: "http"` is strictly better (no shell at all).
   - Recommendation: The planner should flag this to the user. `type: "http"` eliminates the curl script entirely (no HOOK-02 script mount needed, only the settings.json mount). D-09 can be superseded. This is a context-update, not a blocker.

2. **Named volume `/root/.claude/` + read-only settings.json bind mount interaction**
   - What we know: The existing lifecycle.py mounts `/root/.claude` as a named volume (`kubex-creds-{agent_id}`) for credential persistence. Docker resolves bind mounts after volume mounts — the settings.json bind mount at `/root/.claude/settings.json:ro` will shadow the named volume for that specific path.
   - What's unclear: Whether Docker SDK preserves this mount ordering reliably across different Docker Engine versions.
   - Recommendation: Add the settings.json bind mount entry AFTER the credential volume in the `volumes` dict, and add an integration test that verifies the file is read-only despite the parent being a writable volume.

3. **Gateway redis_db0 vs redis_db3 for audit queries**
   - What we know: CLIRuntime writes audit data using its existing Redis connection (`redis.asyncio.from_url("redis://redis:6379")` which defaults to DB 0). Gateway uses `redis_db1` for budget tracking and `redis_db3` for lifecycle events.
   - What's unclear: Which Redis DB the audit sorted sets should live in. DB 0 is what CLIRuntime has access to; Gateway has a `redis_db0` attribute (verify this exists in GatewayService).
   - Recommendation: Audit data lives in DB 0 (same DB as the Broker result keys `task:result:{task_id}`). The Gateway audit endpoint uses `gateway.redis_db0`. Verify this attribute exists in the GatewayService class.

## Key Discovery Summary for Planner

**The most important finding that changes D-09:** Claude Code has a native `type: "http"` hook that directly POSTs the event JSON to any URL. This completely eliminates the need for a curl relay shell script (D-09). The settings.json hook config with `"type": "http", "url": "http://127.0.0.1:8099/hooks"` achieves exactly what D-09 planned, with:
- Zero shell execution surface area (vs curl script)
- No shell script file to generate, validate, or mount
- Pure JSON configuration in the single settings.json file
- No HOOK-02 shell-script read-only mount needed (settings.json read-only mount is sufficient)

The planner should present this to the user as an upgrade to D-09 before locking the plan.
